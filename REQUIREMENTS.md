This is a library for tracing ordering of operations within transactions to
detect potentials for an application to deadlock.

This is only intended for use in test suites and never for use in production.

This extension should be loaded at the start of testing and then the last thing that should be called at the end of testing (or at least, the end of all tests that make use of Prisma), should be the assertNoDeadlockRisk();

It detects two types of deadlock risk:
1. Inconsistent ordering of table locking in transactions
2. Inconsistent ordering of rows updated within a table

The extension patches passthrough methods for all standard operations so that it can record the operation parameters and then passes to the existing query for execution.

### 1. Table Ordering

1.0. The library, when initialised, creates a directed graph for all table locks

1.1. Whenever a transaction is opened, the library tracks the order of table locks within that transaction
1.2. Any operation that locks records (SELECT FOR UPDATE, UPDATE, DELETE, INSERT), is recorded.
1.3. The record of the call must include the first caller in the stacktrace that wasn't a prisma library (ie the origin of the request in the codebase)
1.4. The library adds the record of the call to the directed graph ONLY IF the records have not already been locked by a previous operation in that transaction (eg an UPDATE following a SELECT FOR UPDATE is not recorded if it involves the same records as were selected for update).
That is, once locked in a transaction, a table remains locked and there can be no cycles in the graph for a single transaction
1.5. Table lock order is tracked per-transaction and edges are added to the global graph as operations occur. The global graph accumulates all lock ordering patterns across the entire test suite.
1.6. The library exposes assertConsistentTableLocking() that checks the directed graph for cycles.
1.7. If assertConsistentTableLocking() detects a cycle, it must throw an error, and the error information must include the smallest set of tables in the cycle, and a list of callers involved in creating the cycle (along with which table they cause to be locked)
1.8. The assertConsistentTableLocking() MAY only indicate 1 cycle if there are many, it does not need to enumerate all of them


## 2. Row ordering

2.1. When an update, or delete (or select for update) is called, the library records the following:
- What ordering (if any) was used on the records (note that this is only ever the case for SELECT FOR UPDATE. UPDATE and DELETE operations are, by definition, unordered)
- The caller (first in the stacktrace outside of prisma libraary)
2.2. The record primary keys's impacted are added to a directed graph for that table
2.3. If the updates occur within a transaction, then records are only added to the transaction the first time they are locked - that is within a transaction, there should be no cycles in the directed graph because a record once locked in the transaction remains locked.
2.4. The library exposes the function assertConsistentRowLocking() which checks that locking was always done in a consistent order for rows in each record type (by same order, we mean there were no cycles in the graph)
2.5. The assertConsistentRowLocking() supports specifying which tables to check, and supports a strict optin, which, takes one of the following values:
ASC - The primary keys in the graph must be in ascending order
DESC - The primary keys in the graph must be descending order
true - The primary keys must be in either all ascending, or all descending (on a per table basis, ie it's fine if users is always ascending and posts is always descending)
false - The primary keys may be in any order, provided there are no cycles
2.7. When adding to the graph, it should deduplicate (eg if during tests the same codepath is called multiple times, it should only be added to the graph once)

Row ordering is determined by comparing the IDs returned by the query itself

## 3. Other requirements

3.1. The library must support adding checks to the functions provided by the library in ../prisma-lock-for-update
3.2. When raw queries are executed, the library should make a best effort to interpret the table, but not the records involved. If inference fails, log the error and continue
3.3 Use standard libraries (eg for detecting cycles in graphs), avoid reimplementation of cycle detection
3.4 Graphs should not be reset unless the user explicitly calls for it, as the purpose is to generate a holistic graph of the entire app to ensure the app as a whole does not generate deadlock cycles
3.5 The library should have robust testing


When detecting the first relavent line in the stack trace, it should ignore node_modules and any code generated by prisma.


The library exposes the following:
- Standard Prisma behaviour for adding an extension
- assertConsistentTableLocking()
- assertConsistentRowLocking()
- assertNoDeadlockRisk() - calls both of the above
